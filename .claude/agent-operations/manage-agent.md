# manage-agent

## Purpose

Edit this agent's `identity.md` (purpose, responsibilities, capabilities, constraints, professional lens, sibling-file lookup table). Does NOT create entities, references, or ops — use the matching `create-*` / `manage-*` ops for those.

**Scope (v1 limitation).** This op edits `<agent>/identity.md` only. **Agent-folder rename, archive, and delete are out of scope for v1.** To rename/archive/delete an agent folder, the user does so directly in the filesystem + git (update root `INDEX.md` by hand). A v2 candidate covers `manage-agent rename / archive / delete` with cross-zone path rewriting.

## Invocation

- `/agent manage-agent` — default agent.
- `/agent <name>-agent/ manage-agent` — named agent.
- Args: none. Interactive.

## Inputs

- Current `<agent>/identity.md`.
- User edits proposed in dialog.

## References

- `<agent>/identity.md`.

## Outputs

- Updated `<agent>/identity.md`.

## Steps

1. Read current `identity.md` and present it to the user.
2. Ask which section to edit (or "all").
3. Walk through edits one section at a time. Show old → new before writing.
4. **Lookup table.** The sibling-file lookup table in `identity.md` is generated by `create-agent` from the canonical scaffold and stays in sync via subsequent create-/manage- ops. `manage-agent` only edits it when the user is renaming an entity/reference/op (the corresponding manage-* op writes the actual rename; `manage-agent` updates the lookup-table row to match).
5. Write the updated file.
6. Log.

## Grounding-rule application

- Identity content is user-authored — direct ground-truth. Write through after user confirms each change.
- If the user proposes content that is aspirational, interpretive, or otherwise judgment-requiring (not a direct factual edit), either confirm explicitly inline or park in `<agent>/pending/` per the standard pending-file shape and update `pending/INDEX.md`.
